# Garage Lens Reader API

Version: `0.4.5`

Garage Lens Reader API is a read-only HTTP API that runs beside a self-hosted TeslaMate deployment. It reads TeslaMate PostgreSQL data and returns JSON for the Garage Lens mobile app.

The mobile app does not connect to Tesla, does not connect directly to PostgreSQL, and does not need database credentials. The app only needs:

- Reader API URL, for example `https://reader.example.com`
- Access token, generated by the TeslaMate server owner

## Product Data Flow

```text
Tesla vehicle
  -> Tesla servers
  -> TeslaMate service on the owner's server
  -> TeslaMate PostgreSQL database
  -> Garage Lens Reader API
  -> Garage Lens mobile app
```

TeslaMate remains responsible for collecting Tesla data 24/7. Garage Lens only reads already-collected TeslaMate data through Reader API.

## Security Model

- Do not expose PostgreSQL directly to the public internet.
- Run Reader API on the same server or Docker network as TeslaMate.
- Use a read-only PostgreSQL user when possible.
- Expose only Reader API through HTTPS.
- Protect all data endpoints with `Authorization: Bearer <token>`.
- Rotate the token after public testing or if it has been shared too widely.

`GET /api/health` is intentionally public so the app can verify that the service is alive. All other `/api/*` data endpoints require the bearer token when `READER_API_TOKEN` is configured.

## App Setup

In Garage Lens, the user should enter:

```text
Reader API URL: https://reader.example.com
Access Token: <token provided by the server owner>
```

The app performs:

1. `GET /api/health`
2. `GET /api/cars` with bearer token

If both calls pass, Garage Lens saves the URL/token locally and starts loading vehicle data. If validation fails, vehicle data remains hidden.

## Authentication

Send the token as a bearer token:

```http
Authorization: Bearer YOUR_TOKEN
```

Example:

```bash
curl -H "Authorization: Bearer YOUR_TOKEN" \
  https://reader.example.com/api/cars
```

Unauthorized response:

```json
{
  "error": "unauthorized"
}
```

## Environment Variables

Reader API supports either `DATABASE_URL` or separate PostgreSQL variables.

| Variable | Required | Description |
| --- | --- | --- |
| `PORT` | No | Reader API port. Default: `8787`. |
| `READER_API_TOKEN` | Recommended | Bearer token required by data endpoints. |
| `DATABASE_URL` | Optional | Full PostgreSQL connection string. If set, separate DB host/user variables are ignored. |
| `DATABASE_SSL` | Optional | Set to `true`, `1`, `yes`, or `on` to enable SSL with `rejectUnauthorized: false`. |
| `DATABASE_HOST` | Optional | PostgreSQL host. Default: `127.0.0.1`. |
| `DATABASE_PORT` | Optional | PostgreSQL port. Default: `5432`. |
| `DATABASE_NAME` | Optional | PostgreSQL database. Default: `teslamate`. |
| `DATABASE_USER` | Optional | PostgreSQL user. Default: `teslamate_readonly`. |
| `DATABASE_PASSWORD` | Optional | PostgreSQL password. |

## Endpoints

| Endpoint | Auth | Purpose |
| --- | --- | --- |
| `GET /api/health` | No | Reader status, version, database connection summary. |
| `GET /api/diagnostics/schema` | Yes | TeslaMate schema diagnostics and missing table/column checks. |
| `GET /api/cars` | Yes | List cars available in the TeslaMate database. |
| `GET /api/cars/:carId/overview` | Yes | Main dashboard payload for app home, drives, charges, and insights. |
| `GET /api/cars/:carId/drives?limit=50` | Yes | Recent drives. Max limit: `200`. |
| `GET /api/cars/:carId/drives/:driveId` | Yes | Drive detail with sampled tracking points. |
| `GET /api/cars/:carId/drives/:driveId/tracking` | Yes | Drive route/tracking points only. |
| `GET /api/cars/:carId/charging/sessions?limit=50` | Yes | Recent charging sessions. Max limit: `200`. |
| `GET /api/cars/:carId/charging/sessions/:chargeId` | Yes | Charge detail with charging samples. |
| `GET /api/cars/:carId/visited-map` | Yes | Downsampled route groups for visited/lifetime map. |
| `GET /api/database/info` | Yes | Database size, row counts, first/latest data timestamps. |
| `GET /api/database/settings` | Yes | TeslaMate settings such as unit/range/language/theme. |
| `GET /api/cars/:carId/data-quality` | Yes | Incomplete drives, incomplete charges, missing positions, last healthy sample. |
| `GET /api/cars/:carId/database/tables` | Yes | Vehicle-related raw table catalog. |
| `GET /api/cars/:carId/database/tables/:tableName` | Yes | Paginated raw rows from one supported vehicle-related table. |
| `GET /api/cars/:carId/database/export` | Yes | Limited raw export across supported vehicle-related tables. |

## Main Payload: `GET /api/cars/:carId/overview`

This is the primary endpoint used by the app after URL/token validation.

Top-level shape:

```json
{
  "vehicle": {},
  "monthlyStats": {},
  "drives": [],
  "charges": [],
  "locations": [],
  "database": {},
  "analytics": {}
}
```

### `vehicle`

Common fields:

| Field | Description |
| --- | --- |
| `displayName` | Car name from TeslaMate. |
| `model` | Vehicle model if available. |
| `state` | Derived latest state. |
| `batteryLevel` | Latest battery level. |
| `usableBatteryLevel` | Latest usable battery level. |
| `ratedRangeKm` | Latest rated range in km. |
| `idealRangeKm` | Latest ideal range in km. |
| `odometerKm` | Latest odometer in km. |
| `latitude` / `longitude` | Latest known position. |
| `lastSeen` | Latest position timestamp. |
| `outsideTempC` / `insideTempC` | Latest temperature readings. |
| `powerKw` | Latest power value. |

### `monthlyStats`

| Field | Description |
| --- | --- |
| `distanceKm` | This month's driven distance. |
| `driveCount` | This month's completed drive count. |
| `energyKwh` | Estimated drive energy. |
| `chargeEnergyKwh` | This month's charged energy. |
| `efficiencyWhPerKm` | Estimated driving efficiency. |
| `chargeCount` | This month's charging session count. |
| `chargingCost` | This month's charging cost if TeslaMate has cost data. |
| `onlineHours` | This month's online hours. |
| `asleepHours` | This month's asleep hours. |

### `drives`

Returns up to 10 recent drives in the overview payload.

For more rows, use:

```text
GET /api/cars/:carId/drives?limit=50
```

Drive detail:

```text
GET /api/cars/:carId/drives/:driveId
```

### `charges`

Returns up to 10 recent charging sessions in the overview payload.

For more rows, use:

```text
GET /api/cars/:carId/charging/sessions?limit=50
```

Charge detail:

```text
GET /api/cars/:carId/charging/sessions/:chargeId
```

### `analytics`

The overview analytics payload is designed to support TeslaMate-style charts and insight cards:

| Field | Description |
| --- | --- |
| `currentDrive` | Current drive snapshot. Currently returns neutral values if no active drive is detected. |
| `currentCharge` | Current charge snapshot. Currently returns neutral values if no active charge is detected. |
| `chargingCosts` | Charging energy/cost split, cost per kWh, cost per 100 km, net/gross consumption. |
| `batteryStats` | Estimated capacity, nominal pack estimate, degradation, best/worst range. |
| `dataQuality` | Incomplete drives/charges, missing positions, last healthy timestamp. |
| `amortization` | Placeholder financial fields. Currently neutral values unless implemented later. |
| `stateTimeline` | State distribution over time. |
| `monthlyMileage` | Monthly mileage chart data. |
| `rangeDegradation` | Range degradation chart data. |
| `chargingCurves` | Charging power / battery-level curve data. |
| `speedRates` | Efficiency by speed data. |
| `speedTemperature` | Efficiency by outside temperature data. |
| `topStations` | Charging station energy/cost/session summary. |

## Raw Vehicle Tables

Reader API can expose vehicle-related raw data for debugging and advanced app features.

Start with:

```text
GET /api/cars/:carId/database/tables
```

This returns supported tables, columns, row counts, date columns, and date ranges.

Then query one table:

```text
GET /api/cars/:carId/database/tables/:tableName?limit=200&offset=0&order=desc
```

Supported query parameters:

| Parameter | Default | Max | Description |
| --- | --- | --- | --- |
| `limit` | `200` | `1000` | Rows per page. |
| `offset` | `0` | - | Pagination offset. |
| `order` | `desc` | - | `asc` or `desc`. |
| `from` | - | - | ISO date lower bound, if the table has a date column. |
| `to` | - | - | ISO date upper bound, if the table has a date column. |

Reader only exposes tables that are related to the selected vehicle:

- selected row from `cars`
- direct `car_id` tables such as `positions`, `drives`, `states`, `charging_processes`
- `charges` through `charging_processes.car_id`
- referenced `addresses`
- referenced `geofences`
- compatible future TeslaMate tables with `car_id`

## Runtime Logs

Reader API writes API request logs and PostgreSQL query logs to stdout/stderr. In Docker, they can be viewed with:

```bash
docker logs -f reader-api-reader-1
```

If using the compose file from the Reader package:

```bash
docker compose --env-file .env.reader -f docker-compose.reader.yml logs -f reader
```

Logs are one-line JSON records.

API request example:

```json
{"ts":"2026-06-21T10:00:00.000Z","level":"info","event":"api_request_started","requestId":"...","method":"GET","path":"/api/cars/1/overview","query":{}}
{"ts":"2026-06-21T10:00:02.120Z","level":"info","event":"api_request_completed","requestId":"...","method":"GET","path":"/api/cars/1/overview","statusCode":200,"durationMs":2120}
```

Database query example:

```json
{"ts":"2026-06-21T10:00:00.120Z","level":"info","event":"db_query_started","requestId":"...","queryId":"...","sql":"select * from cars where id = $1","params":[1]}
{"ts":"2026-06-21T10:00:00.132Z","level":"info","event":"db_query_completed","requestId":"...","queryId":"...","durationMs":12,"rowCount":1}
```

When troubleshooting, match `requestId` across `api_request_completed` and `db_query_*` logs. This shows which SQL queries were triggered by a specific app request and how long each query took.

Reader does not log bearer tokens or database passwords.

### `401 unauthorized`

The token is missing or does not match `READER_API_TOKEN`.

Check:

- App token field
- Reverse proxy header forwarding
- Reader container environment variable

### `reader_api_error`

Reader API caught an internal error.

Check:

- Reader container logs
- PostgreSQL connection variables
- TeslaMate schema compatibility
- `GET /api/diagnostics/schema`

### Health is OK but app still cannot show data

Check:

1. `GET /api/cars` with token returns at least one car.
2. `GET /api/cars/1/overview` returns JSON.
3. The app URL includes scheme, for example `https://reader.example.com`.
4. HTTPS certificate is valid if accessed from a real phone.

## Recommended Deployment Pattern

Reader API should be deployed as a sidecar/add-on service beside TeslaMate.

```text
TeslaMate Docker network
  - teslamate
  - database/postgres
  - grafana
  - garage-lens-reader-api
```

Uninstalling Reader API should not affect TeslaMate because Reader only performs read queries and does not modify TeslaMate tables.
